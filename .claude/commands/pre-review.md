Run an Aluci label pre-review of this repository and write the notes to
`pre-review-notes.md`.

## Steps

1. Map the repository: read the full directory tree.
2. Read key files: `LICENSE`, `README.md`, `AI_USAGE.md`, and any docs/ or
   architecture files.
3. Inspect dependency files (package.json, Cargo.toml, pyproject.toml, go.mod,
   CMakeLists.txt, etc.) and lockfiles.
4. Look for setup automation: scripts, Makefile, Dockerfile, docker-compose,
   devcontainer.json.
5. Scan a representative sample of source files for: code quality signals,
   telemetry/tracking, resource efficiency patterns.
6. Check for CI/CD config, test infrastructure, PR templates, CODEOWNERS.
7. For each Aluci criterion (C1–C5), collect your observations.
8. Write `pre-review-notes.md` at the repo root using the format below.

## Output format for `pre-review-notes.md`

```markdown
# Pre-Review Notes: [PROJECT_NAME]

**Repository:** [URL or path]
**Pre-review date:** [YYYY-MM-DD]
**Generated by:** Claude Code (pre-review assistant)

> These are automated observations to assist the Aluci labeling committee.
> They are NOT a final verdict. Human reviewers should verify all findings
> and use their judgment for the final report.

---

## 1. Open Source & Licensing

**What I found:**
[Factual observations: license file location, license type, any discrepancies with badges or package metadata.]

**⚠️ Points of attention:**
[Anything reviewers should look at more closely. Leave empty if nothing to flag.]

**❓ Questions for the applicant:**
[Things that can't be determined from the code alone.]

---

## 2. Technical Excellence & Justification

**What I found:**
[Directory structure summary, dependency tooling, lockfile presence, setup automation, observed patterns.]

**⚠️ Points of attention:**
[Structural issues, missing lockfiles, signs of vibe coding, unjustified tech choices, dead code.]

**❓ Questions for the applicant:**
[E.g., "What motivated the choice of X over Y for this use case?"]

---

## 3. Documentation & Setup Standards

**What I found:**
[README sections present, requirements listed, maturity warnings.]

**⚠️ Points of attention:**
[Missing sections, vague setup instructions, no requirements list.]

**❓ Questions for the applicant:**
[E.g., "The README doesn't mention minimum OS version, is this intentional?"]

---

## 4. AI Usage Transparency

**What I found:**
[AI_USAGE.md presence, tools listed, human validation statement.]

**⚠️ Points of attention:**
[Missing file, incomplete disclosure, no validation statement.]

**❓ Questions for the applicant:**
[E.g., "AI_USAGE.md mentions Copilot but not which modules were AI-assisted."]

---

## 5. Ethical & Human-Centric Design

**What I found:**
[Telemetry/tracking code observed, algorithm choices, third-party calls, data handling.]

**⚠️ Points of attention:**
[Unnecessary data collection, heavy resource usage without justification, privacy gaps.]

**❓ Questions for the applicant:**
[E.g., "The app sends analytics to X, is this documented for end users?"]

---

## Supporting observations

[Anything else that caught your attention: CI/CD setup, test coverage, code review practices, quality gates, things that impressed you, or things that felt off but don't fit neatly into a criterion.]

---

## Suggested recommendations for the committee report

[Draft 2–4 concrete, constructive recommendations the committee could include in their final report. Frame these as improvements, not criticisms.]
```

## Rules

- Do NOT modify any existing project files. Only create `pre-review-notes.md`.
- Cite file paths as evidence for every observation.
- Use a collegial tone: you're helping your teammates, not grading a student.
- If you can't determine something, put it in "Questions for the applicant"
  rather than guessing.
- It's fine to note things that are well done, positive signals matter too.
